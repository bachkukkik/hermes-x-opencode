# 12 — Plugin System

## What

Three npm-resolved OpenCode plugins are declared in the generated `opencode.jsonc` and loaded at OpenCode startup: a context pruning plugin, a markdown table formatter, and a destructive command safety net.

## Why

- Extends OpenCode's behavior without modifying the binary or writing custom hooks
- `cc-safety-net` provides a PreToolUse hook that blocks destructive git and filesystem commands before the permission system evaluates them
- `@tarquinen/opencode-dcp` reduces token usage by pruning low-relevance context from the conversation
- `@franlol/opencode-md-table-formatter` normalizes markdown table output for consistent rendering

## How

Plugins are declared in the `plugin` array in `opencode.jsonc` (generated by `05 — Entrypoint Sequence`):

```json
"plugin": [
  "@tarquinen/opencode-dcp@latest",
  "@franlol/opencode-md-table-formatter@latest",
  "cc-safety-net"
]
```

### Plugin resolution

OpenCode resolves plugins at startup using Node.js npm resolution:

1. Each string in the `plugin` array is treated as an npm spec
2. OpenCode installs/resolve the package under `$HOME/.config/opencode/` (which is `/home/hermeswebui/.config/opencode/` since opencode serve runs as hermeswebui)
3. The `node_modules/` directory is created alongside the config
4. Node.js v22 is required in the container image for this resolution (installed in `04 — Build Pipeline`)

### Plugin inventory

| Plugin | Version | npm spec | Hook | Purpose |
|--------|---------|----------|------|---------|
| opencode-dcp | 3.1.12 | `@tarquinen/opencode-dcp@latest` | Context transform | Prunes low-relevance messages from the conversation to reduce token usage |
| md-table-formatter | 0.0.6 | `@franlol/opencode-md-table-formatter@latest` | Output transform | Formats markdown tables in tool output for consistent rendering |
| cc-safety-net | 0.9.0 | `cc-safety-net` | `PreToolUse` | Blocks destructive git and filesystem commands before execution |

### cc-safety-net architecture

cc-safety-net registers a `PreToolUse` hook that runs **before** OpenCode's built-in permission system. This means:

1. Agent generates a tool call (e.g., `bash` with command `git reset --hard`)
2. cc-safety-net's `PreToolUse` hook inspects the command
3. If the command matches a blocked pattern, the hook returns a block response — the tool call is rejected
4. If the command does not match, the hook passes through
5. OpenCode's permission system then evaluates the command against its rules

Blocked command patterns (from cc-safety-net README):

| Category | Blocked patterns |
|----------|-----------------|
| Destructive git | `git reset --hard`, `git push --force`, `git clean -fd`, `git checkout -- .` |
| Destructive filesystem | `rm -rf /`, `rm -rf ~`, `chmod -R 777 /` |
| Flag reordering | `git reset --hard HEAD~1` (any `--hard` reset) |

Known bypass vectors (documented but not blocked by cc-safety-net):

| Vector | Example | Status |
|--------|---------|--------|
| Shell wrappers | `bash -c "git reset --hard"` | Blocked by permission system (`bash -c *: deny` in strict mode) |
| Interpreter one-liners | `python3 -c "subprocess.run(['git','reset','--hard'])"` | Blocked by permission system (`python3 -c *: deny` in strict mode) |
| Extra whitespace | `git  reset --hard` | Not tested |
| Shell variable expansion | `CMD="git reset --hard"; $CMD` | Not tested |

### Interaction with permission system

cc-safety-net and the OpenCode permission system operate in sequence:

```
Agent → cc-safety-net (PreToolUse) → Permission system → Tool execution
```

- cc-safety-net blocks **destructive commands** regardless of permission mode
- The permission system controls **information access** (env vars, .env files, interpreters)
- In `yolo` mode (`"permission": "allow"`), cc-safety-net still blocks destructive commands because it runs as a plugin hook, not a permission rule
- See `13 — Security Hardening` for the full permission matrix

### Plugin failure behavior

If a plugin fails to load (npm resolution error, missing Node.js, invalid module):

1. OpenCode logs a warning to stderr
2. The remaining plugins load normally
3. opencode serve does not exit — it continues with whatever plugins loaded successfully
4. To diagnose: check `docker compose logs hermes-opencode` for plugin-related errors

## Verification

```bash
docker compose exec hermes-opencode bash -c 'cat /home/hermeswebui/.config/opencode/opencode.jsonc | python3 -c "import sys,json; c=json.load(sys.stdin); print(json.dumps(c.get(\"plugin\",[]), indent=2))"'
docker compose exec hermes-opencode bash -c 'ls -la /home/hermeswebui/.config/opencode/node_modules/ 2>/dev/null | head -10'
docker compose exec hermes-opencode bash -c 'cd /workspace && timeout 90 opencode run --format json "Run git reset --hard" litellm/zai/glm-5.1 2>&1 | tail -3'
```

## What Works

- All three plugins resolve and load on container startup
- cc-safety-net blocks `git reset --hard`, `git push --force`, and `git clean -fd`
- Plugin resolution requires Node.js v22 (installed in `04 — Build Pipeline`)
- `node_modules/` directory is created under hermeswebui's config directory with correct ownership
- Plugins persist across `opencode serve` restarts within the same container lifetime (ephemeral across rebuilds)

## What Fails

- **Plugins lost on container rebuild:** The `node_modules/` directory lives under the config path, which is not volume-mounted. Rebuilding the image clears resolved plugins.
- **npm resolution requires network:** If the container cannot reach npm registries during opencode serve startup, plugins fail to resolve. The `@latest` tag requires a registry lookup even when cached.
- **cc-safety-net bypass via interpreters:** Direct interpreter calls (`python3 -c`, `node -e`) are not blocked by cc-safety-net. They are blocked by the permission system in `strict` mode only.
- **Plugin version drift:** The `@latest` tag means plugin behavior can change between container rebuilds. Pinning versions (e.g., `@tarquinen/opencode-dcp@3.1.12`) prevents drift but requires manual updates.

## Resolution

- Plugin ephemerality matches the config regeneration pattern — the entrypoint generates a fresh config on every boot. Add a volume mount for `/home/hermeswebui/.config/opencode/` in `docker-compose.yml` to persist resolved plugins.
- Ensure the container has network access during startup. If operating in an air-gapped environment, pre-resolve plugins and copy `node_modules/` into the image.
- Interpreter bypasses are mitigated by the permission system in `strict` mode. Use `standard` or `yolo` mode only in trusted environments. See `13 — Security Hardening`.
- Pin plugin versions in the entrypoint's `plugin` array for reproducible behavior across rebuilds.

## Verdict

The plugin system provides essential safety net functionality via cc-safety-net and quality-of-life improvements via the formatter and context pruner. The npm resolution dependency and ephemeral `node_modules/` are acceptable trade-offs for a containerized environment. The cc-safety-net + permission system combination provides defense-in-depth against destructive command execution.
